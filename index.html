🚀 What is Apex? And What Are Data Types?

🌍 Real-world analogy
Think of Salesforce as a giant, pre-built house. You live in it comfortably — CRM, contacts, accounts, leads — all there. But what if you want to add a secret room? Or make the door open only when it rains? Apex is the custom builder you hire to do exactly that. It's the programming language that lets you add custom logic inside Salesforce.

Apex is a strongly-typed, object-oriented programming language created by Salesforce. It runs on Salesforce's servers, not on your computer. That's why you don't need to install anything — you just write it in the Salesforce Developer Console and it runs in the cloud.

📦 Data Types — "The kind of box you use to store something"

Every piece of information in Apex needs to be stored in a variable. A variable is just a named box. The data type tells Apex what kind of thing the box can hold — numbers, text, yes/no answers, etc.

🗄️ What Are sObjects?

🌍 Real-world analogy
Imagine Salesforce is a big warehouse. Every shelf in that warehouse holds a type of record — one shelf for Accounts, one for Contacts, one for Opportunities. An sObject is a single row from one of those shelves. The "s" stands for Salesforce Object. When you want to work with a Salesforce record in Apex, you load it into an sObject variable.

Salesforce stores data in objects (like database tables). Standard objects include Account, Contact, Lead, Opportunity. You can also create custom objects (e.g., Student__c). In Apex, every record is represented as an sObject.

📌 Creating an sObject Variable

sObjectDemo.apex
// ─────────────────────────────────────────────
// WAY 1: Create and set fields one by one
// ─────────────────────────────────────────────
Account acc1 = new Account();     // Create an empty Account sObject
acc1.Name    = 'Tata Motors';      // Set the Name field
acc1.Phone   = '9999988888';      // Set the Phone field
acc1.Industry = 'Automotive';     // Set the Industry field
System.debug('Account: ' + acc1);

// ─────────────────────────────────────────────
// WAY 2: Set fields right when you create (inline)
// Cleaner and shorter — preferred way!
// ─────────────────────────────────────────────
Account acc2 = new Account(
    Name     = 'Reliance Industries',
    Phone    = '8888877777',
    Industry = 'Energy'
);
System.debug('Account 2: ' + acc2.Name);

// ─────────────────────────────────────────────
// WAY 3: Working with Contact (has a relationship to Account)
// ─────────────────────────────────────────────
Contact c = new Contact(
    FirstName = 'Priya',
    LastName  = 'Mehta',
    Email     = 'priya@example.com'
);
System.debug('Contact: ' + c.FirstName + ' ' + c.LastName);

// ─────────────────────────────────────────────
// WAY 4: Generic sObject (when you don't know the type at design time)
// Think of it as a "general purpose" container
// ─────────────────────────────────────────────
sObject genericObj = new Account(Name = 'Generic Corp');
System.debug('Generic: ' + genericObj.get('Name'));  // Must use .get()

// ─────────────────────────────────────────────
// WAY 5: Querying an existing record from database (SOQL)
// Read it as: "SELECT these fields FROM this object WHERE condition"
// ─────────────────────────────────────────────
Account existingAcc = [
    SELECT Id, Name, Phone
    FROM   Account
    WHERE  Name = 'Tata Motors'
    LIMIT  1
];
System.debug('Found: ' + existingAcc.Name + ' | ID: ' + existingAcc.Id);

💾 DML Statements — Saving, Updating & Deleting Records

🌍 Real-world analogy
You've filled out a form (sObject). Now DML (Data Manipulation Language) is the post office. insert = mailing a new application. update = sending a correction. delete = shredding the form. upsert = "update if it exists, otherwise create a new one."

🔵 The 5 DML Operations

DML_Operations.apex
// ─────────────────────────────────────────────
// 1. INSERT — Create a brand new record
// ─────────────────────────────────────────────
Account newAcc = new Account(Name = 'Infosys Ltd', Phone = '7777766666');
insert newAcc;    // Saves to database. After this, newAcc.Id is populated!
System.debug('New Account ID: ' + newAcc.Id);

// ─────────────────────────────────────────────
// 2. UPDATE — Modify an existing record
// You MUST have the Id to update
// ─────────────────────────────────────────────
newAcc.Phone = '1234567890';   // Change the phone number
update newAcc;                 // Push the change to database
System.debug('Updated Phone: ' + newAcc.Phone);

// ─────────────────────────────────────────────
// 3. DELETE — Remove a record permanently
// ─────────────────────────────────────────────
Account toDelete = [SELECT Id FROM Account WHERE Name = 'OldCompany' LIMIT 1];
delete toDelete;
System.debug('Deleted successfully');

// ─────────────────────────────────────────────
// 4. UPSERT — Update OR Insert (whichever applies)
// If a matching record exists → update it
// If no match found          → create a new one
// ─────────────────────────────────────────────
Account maybeNew = new Account(Name = 'Wipro', Phone = '5555544444');
upsert maybeNew;   // If Wipro exists, update; otherwise create
System.debug('Upserted: ' + maybeNew.Id);

// ─────────────────────────────────────────────
// 5. UNDELETE — Restore a deleted record from the Recycle Bin
// ─────────────────────────────────────────────
Account restored = [SELECT Id FROM Account WHERE Name = 'OldCompany'
                     ALL ROWS];   // ALL ROWS includes deleted records
undelete restored;
System.debug('Restored!');

🟡 Bulk DML — Always Do It This Way!

🌍 Real-world analogy
Imagine you need to send 200 parcels. Would you go to the post office 200 times (once per parcel)? No! You'd take all 200 in one trip. Bulk DML = one trip to the post office for all records. Salesforce allows max 150 DML statements per transaction — so always batch your records into a list!
❌ Bad — DML inside a loop
for(Integer i=0; i<10; i++) {
  Account a = new Account(
    Name='Acc'+i);
  insert a; // 10 DML calls!
}
✅ Good — Bulk DML with List
List<Account> accs =
  new List<Account>();
for(Integer i=0; i<10; i++) {
  accs.add(new Account(Name='Acc'+i));
}
insert accs; // Only 1 DML call!
BulkDML.apex
// Create a list to hold all new accounts
List<Account> accountList = new List<Account>();

// Loop and build the list — do NOT insert inside the loop!
for (Integer i = 1; i <= 5; i++) {
    accountList.add(new Account(
        Name    = 'Company ' + i,
        Phone   = '900000000' + i,
        Industry = 'Technology'
    ));
}

// One single DML call inserts ALL 5 accounts at once — EFFICIENT!
insert accountList;

// After insert, each item in the list now has its Id populated
for (Account a : accountList) {
    System.debug('Inserted: ' + a.Name + ' → ID: ' + a.Id);
}
System.debug('Total DML statements used: 1 (bulk!)');

🔴 DML Exception Handling — Always Be Prepared!

DMLException.apex
// What if the DML fails? (e.g., required field missing, duplicate, etc.)
// You need try-catch to handle it gracefully instead of crashing

Account badAccount = new Account();   // Name is REQUIRED but we're not setting it!

try {
    // Try to insert — this will FAIL because Name is required
    insert badAccount;
    System.debug('Insert worked!');    // This line will NOT run

} catch (DmlException e) {
    // This block runs when something goes wrong with DML
    System.debug('DML Failed!');
    System.debug('Error Message: ' + e.getMessage());
    System.debug('Which field: '  + e.getDmlFieldNames(0));
    System.debug('Status code: '  + e.getDmlStatusCode(0));

} catch (Exception e) {
    // Generic catch — catches ANY other kind of error
    System.debug('Unexpected error: ' + e.getMessage());

} finally {
    // This block ALWAYS runs, whether error or not
    // Good for cleanup code
    System.debug('DML attempt finished.');
}

🟣 Database Methods — More Control Than Plain DML

🌍 Real-world analogy
Regular DML is like submitting a batch of forms — if one form has an error, the whole batch is rejected. Database methods are like submitting them one-by-one with a form checker: even if 2 fail, the other 8 still get processed. You get a detailed report of what succeeded and what failed.
DatabaseMethods.apex
List<Account> accs = new List<Account>{
    new Account(Name = 'Valid Company'),    // ✅ Will succeed
    new Account(),                          // ❌ Will fail — no Name!
    new Account(Name = 'Another Valid')     // ✅ Will succeed
};

// Database.insert(list, allOrNone)
// allOrNone = false → even if some fail, save the rest
// allOrNone = true  → if ANY fail, roll back ALL (like plain DML)
Database.SaveResult[] results = Database.insert(accs, false);

// Loop through results to see what happened to each record
for (Integer i = 0; i < results.size(); i++) {
    Database.SaveResult sr = results[i];

    if (sr.isSuccess()) {
        System.debug('Row ' + i + ' SAVED. ID: ' + sr.getId());
    } else {
        for (Database.Error err : sr.getErrors()) {
            System.debug('Row ' + i + ' FAILED: ' + err.getMessage());
            System.debug('Fields: ' + err.getFields());
        }
    }
}
OUTPUT
Row 0 SAVED. ID: 0012w000001AAAA
Row 1 FAILED: Required fields are missing: [Name]
Fields: (Name)
Row 2 SAVED. ID: 0012w000001BBBB

📌 Static Variables & Methods

🌍 Real-world analogy
Imagine a school. Each classroom has its own whiteboard (instance variable — belongs to that room). But there's one announcement board in the hallway that everyone in the school shares (static variable — belongs to the class itself). If one student writes on the hallway board, every student in every classroom sees it.

A static variable/method belongs to the class, not to any specific object. You don't need to create an instance (object) to use it — you just call it using the class name directly.

StaticDemo.apex
public class OrderProcessor {

    // STATIC variable — shared across ALL instances of this class
    // Great for counting how many times something happened in a transaction
    // Prevents trigger recursion (very common real-world use!)
    public static Integer orderCount = 0;

    // STATIC flag — used to prevent recursive trigger calls
    public static Boolean isRunning = false;

    // INSTANCE variable — each OrderProcessor object has its own copy
    public String processorName;

    // STATIC METHOD — can call without creating an object
    // Note: static methods can ONLY use static variables, not instance variables
    public static void processOrder(String itemName) {
        if (!isRunning) {
            isRunning = true;
            orderCount++;   // Increment shared counter
            System.debug('Processing order #' + orderCount + ': ' + itemName);
            isRunning = false;
        } else {
            System.debug('Already running — skipping recursive call');
        }
    }

    // STATIC method — utility / helper (no object needed)
    public static String formatPrice(Decimal amount) {
        return '₹' + amount.setScale(2);  // Always 2 decimal places
    }

    // INSTANCE method — needs an object to call
    public void introduce() {
        System.debug('I am processor: ' + processorName);
    }
}
UsingStaticDemo.apex
// Call static method — NO object needed, just use class name!

System.debug(formatted);  // ₹1999.50

// Instance method NEEDS an object

p1.processorName = 'Processor-A';

🔧 Static Initialization Block

A static initializer runs once when the class is first loaded — perfect for one-time setup like loading configuration data.

public class Config {

    public static Map<String, String> settings;

        settings.put('MAX_ROWS', '200');

🔒 Apex Properties — Smart Variable Guards

🌍 Real-world analogy
Think of a property as a security-checked locker. Instead of anyone being able to open the locker directly, you go through a receptionist (getter) who retrieves it for you, or a checker (setter) who validates before putting something in. This protects your data from being set to invalid values.

A property wraps a private variable with get and set accessors. This is a core OOP concept called encapsulation — hide the internals, expose a clean interface.

PropertiesDemo.apex
public class BankAccount {

    // Private variable — nobody outside can touch this directly
    private Decimal balance;

    // ─────────────────────────────────────────────
    // FULL PROPERTY — get and set with logic
    // ─────────────────────────────────────────────
    public Decimal Balance {
        get {
            // Logic before returning the value
            System.debug('Balance was accessed');
            return balance;
        }
        set {
            // 'value' is the keyword for what's being assigned
            if (value < 0) {
                // Guard: reject negative balances
                throw new IllegalArgumentException('Balance cannot be negative!');
            }
            balance = value;
        }
    }

    // ─────────────────────────────────────────────
    // READ-ONLY PROPERTY — only has get, no set
    // Outside code can READ it but NOT change it
    // ─────────────────────────────────────────────
    public Boolean IsRich {
        get { return balance > 100000; }
        // No 'set' block — this is READ-ONLY
    }

    // ─────────────────────────────────────────────
    // WRITE-ONLY PROPERTY — rarely used, but valid
    // Someone can set a PIN but cannot read it back
    // ─────────────────────────────────────────────
    private String secretPin;
    public String PIN {
        set {
            secretPin = value.toUpperCase();   // Always store in uppercase
        }
        // No 'get' — write-only!
    }
}

// ─────────────────────────────────────────────
// AUTOMATIC PROPERTY — shorthand, no custom logic needed
// Apex creates the private backing variable for you automatically
// ─────────────────────────────────────────────
public class Person {
    // Instead of writing get { return name; } set { name = value; }
    // Automatic property does it in ONE LINE:
    public String Name  { get; set; }
    public Integer Age  { get; set; }
    public String Email { get; private set; }  // Public read, private write
}

🧬 Extending a Class — Inheritance

🌍 Real-world analogy
A Vehicle can move, has a fuel tank, and makes a sound. A Car is a Vehicle, but it also has a trunk and a stereo. A Truck is also a Vehicle, but it has a cargo bay. Inheritance means Car and Truck inherit everything from Vehicle and then add their own extras. You don't rewrite "move" for every vehicle — you write it once in Vehicle.
InheritanceDemo.apex
// ─────────────────────────────────────────────
// PARENT CLASS (also called Base Class or Super Class)
// virtual = this class CAN be extended (subclassed)
// ─────────────────────────────────────────────
public virtual class Animal {

    public String name;
    public Integer age;

    // Constructor — sets up the object when you create it
    public Animal(String n, Integer a) {
        name = n;
        age  = a;
    }

    // virtual method — child classes CAN override this
    public virtual String speak() {
        return name + ' makes a sound';
    }

    // Regular method — child classes INHERIT this as-is
    public void eat() {
        System.debug(name + ' is eating 🍽️');
    }
}

// ─────────────────────────────────────────────
// CHILD CLASS (also called Subclass or Derived Class)
// extends Animal = Dog IS-A Animal, inherits everything
// ─────────────────────────────────────────────
public class Dog extends Animal {

    public String breed;

    // super() calls the PARENT class constructor
    public Dog(String n, Integer a, String b) {
        super(n, a);   // Run Animal's constructor first
        breed = b;
    }

    // override = replace the parent's version with our own
    // Dog has a specific sound — woof!
    public override String speak() {
        return name + ' says: Woof! 🐶';
    }

    // New method only Dog has
    public void fetch() {
        System.debug(name + ' the ' + breed + ' fetches the ball! 🎾');
    }
}

// ─────────────────────────────────────────────
// ANOTHER CHILD — Cat also extends Animal
// ─────────────────────────────────────────────
public class Cat extends Animal {
    public Cat(String n, Integer a) {
        super(n, a);
    }
    public override String speak() {
        return name + ' says: Meow! 🐱';
    }
}
UsingInheritance.apex
Dog dog = new Dog('Bruno', 3, 'Labrador');
Cat cat = new Cat('Whiskers', 5);

// Dog has Animal's methods (inherited)
dog.eat();                        // Inherited from Animal

// Dog's overridden speak() (not Animal's version)
System.debug(dog.speak());       // Bruno says: Woof!

// Dog's own method
dog.fetch();

// Cat's speak
System.debug(cat.speak());       // Whiskers says: Meow!

// POLYMORPHISM — treating different subclasses as their parent type
// We can store Dog and Cat in an Animal variable
// Each one calls ITS OWN speak() method — this is polymorphism!
List<Animal> animals = new List<Animal>{ dog, cat };
for (Animal a : animals) {
    System.debug(a.speak());      // Each animal speaks in its own way!
}

📌 Key OOP Terms decoded:
Inheritance = Child gets everything from parent automatically.
Override = Child replaces parent's method with its own version.
super() = Explicitly call the parent's constructor or method.
Polymorphism = Same method name, different behaviour depending on object type.

🔴 Abstract Classes — A Half-Defined Blueprint

AbstractDemo.apex
// abstract class = a template you CANNOT instantiate directly
// It forces child classes to provide their own implementation
// Use it when: the parent has SOME common logic, but certain methods
// MUST be customised by each child

public abstract class Shape {

    public String color;

    // Abstract method = no body here — child MUST implement it
    public abstract Double calculateArea();

    // Regular method — available to all children
    public void describe() {
        System.debug('I am a ' + color + ' shape. Area: ' + calculateArea());
    }
}

public class Circle extends Shape {
    public Double radius;
    public Circle(Double r, String c) { radius=r; color=c; }

    // MUST implement calculateArea or Apex gives a compile error!
    public override Double calculateArea() {
        return Math.PI * radius * radius;
    }
}

public class Rectangle extends Shape {
    public Double width, height;
    public Rectangle(Double w, Double h, String c) {
        width=w; height=h; color=c;
    }
    public override Double calculateArea() {
        return width * height;
    }
}

// Usage

// INTERFACE — a pure contract
// Only method signatures (name, params, return type) — NO bodies

    public Integer hoursWorked;

        hoursWorked  = hours;

    public String getPaymentSummary() {
        return 'Freelance: ' + hoursWorked + 'hrs × ₹' + hourlyRate
               + ' = ₹' + calculatePayment();
    }

    // Implementing Taxable's calculateTax()
    public Decimal calculateTax(Decimal amount) {

    public Decimal bonus;

        return monthlySalary + bonus;

    public String getPaymentSummary() {
        return 'Salary ₹' + monthlySalary + ' + Bonus ₹' + bonus
               + ' = ₹' + calculatePayment();
    }
}

🛡️ With Sharing vs Without Sharing

🌍 Real-world analogy
Your company has records marked "Manager-only."
with sharing = The Apex class behaves like a normal employee. It respects the sharing rules. If you're not a manager, you can't see manager-only records.
without sharing = The Apex class acts like a system admin — it ignores sharing rules and can see everything.
inherited sharing = The class borrows whatever sharing context it was called from.
SharingDemo.apex
// ─────────────────────────────────────────────
// WITH SHARING
// Enforces the current user's sharing rules (OWD, role hierarchy, sharing rules)
// A user only sees records they are allowed to see
// USE THIS as the default — safest option!
// ─────────────────────────────────────────────
public with sharing class CustomerService {

    public static List<Account> getMyAccounts() {
        // If logged-in user can only see 5 accounts (sharing rules),
        // this query returns only those 5 — NOT all 1000 in the org!

}

// ─────────────────────────────────────────────
// WITHOUT SHARING
// Ignores the user's sharing rules — runs in system mode
// USE THIS only when absolutely necessary:
// e.g., batch jobs, integrations, admin utilities
// NEVER expose this to untrusted users!
// ─────────────────────────────────────────────
public without sharing class AdminService {

    public static List<Account> getAllAccounts() {
        // Returns ALL accounts regardless of who is logged in
        // Even records the user has no permission to see!
        return [SELECT Id, Name FROM Account];
    }
}

// INHERITED SHARING
// Inherits sharing context from the calling class
// If called by a "with sharing" class → enforces sharing

// Best for utility classes used in both contexts
// ─────────────────────────────────────────────
public inherited sharing class UtilityService {

    public static List<Contact> getContacts() {

LimitBestPractices.apex
// ❌ BAD PATTERN — SOQL inside a loop
// If there are 100 accounts, this runs 100 SOQL queries!
// Will CRASH with "Too many SOQL queries: 101" error
List<Account> accounts = [SELECT Id FROM Account];
for(Account a : accounts) {
    // DANGER! Query inside a loop!
    List<Contact> contacts = [SELECT Id FROM Contact
                               WHERE AccountId = :a.Id];
}

// ✅ GOOD PATTERN — One query outside the loop
// Get ALL contacts for ALL accounts in ONE query
// Uses a Map to look up contacts by Account ID
List<Account> accs = [SELECT Id, Name FROM Account];

// Collect all Account IDs into a Set
Set<Id> accIds = new Map<Id, Account>(accs).keySet();

// ONE query to get all related contacts
List<Contact> allContacts = [SELECT Id, FirstName, AccountId
                              FROM Contact
                              WHERE AccountId IN :accIds];

// Build a map: accountId → list of contacts
Map<Id, List<Contact>> accToContacts = new Map<Id, List<Contact>>();
for(Contact c : allContacts) {
    if(!accToContacts.containsKey(c.AccountId)) {
        accToContacts.put(c.AccountId, new List<Contact>());
    }
    accToContacts.get(c.AccountId).add(c);
}

// Now loop — no SOQL inside!
for(Account a : accs) {
    List<Contact> contacts = accToContacts.get(a.Id);
    System.debug(a.Name + ' has ' + (contacts != null ? contacts.size() : 0) + ' contacts');
}
// Total SOQL: 2 (one for accounts, one for contacts) — not 100+!
📌 The Golden Rules of Governor Limits:
1. Never put SOQL inside a loop — query once, iterate over results.
2. Never put DML inside a loop — collect records in a List, DML once.
3. Use Maps for lookups — faster than looping and avoids extra queries.
4. Use Limits class to monitor usage during development.
5. Use Batch Apex for processing more than 50K records.

🧠 The Full OOP Picture — Everything You Actually Need

Object-Oriented Programming (OOP) sounds fancy, but it's really just a way of organising code to mirror the real world. Let's cover every concept you'll encounter as a Salesforce developer — from constructors to polymorphism to access modifiers.

🌍 The Master Analogy
Think of a class as a blueprint for a house. The blueprint describes what a house looks like, what rooms it has, what rules apply. An object is an actual house built from that blueprint. You can build 100 houses from the same blueprint — each one is independent, but they all follow the same design.

🏗️ 1. Constructors — The "Birth" of an Object

A constructor is a special method that runs automatically when you create an object with new. Its job is to set the object up with initial values. It always has the same name as the class and no return type.

ConstructorDemo.apex
public class Student {

    public String  name;
    public Integer age;
    public String  grade;

    // ─────────────────────────────────────────────
    // DEFAULT CONSTRUCTOR — no parameters
    // Runs when you do: new Student()
    // Sets safe default values
    // ─────────────────────────────────────────────
    public Student() {
        name  = 'Unknown';
        age   = 0;
        grade = 'N/A';
        System.debug('Default constructor called');
    }

    // ─────────────────────────────────────────────
    // PARAMETERIZED CONSTRUCTOR — takes arguments
    // Runs when you do: new Student('Rahul', 20, 'A')
    // ─────────────────────────────────────────────
    public Student(String n, Integer a, String g) {
        name  = n;
        age   = a;
        grade = g;
        System.debug('Parameterized constructor called');
    }

    // ─────────────────────────────────────────────
    // ANOTHER CONSTRUCTOR — partial info (name only)
    // This is CONSTRUCTOR OVERLOADING — same class,
    // different constructors with different signatures
    // ─────────────────────────────────────────────
    public Student(String n) {
        name  = n;
        age   = 18;      // sensible default age
        grade = 'Pending';
        System.debug('Name-only constructor called');
    }

    public void display() {
        System.debug('Student → Name: ' + name
                   + ' | Age: ' + age
                   + ' | Grade: ' + grade);
    }
}

🔑 2. The this Keyword — "I am talking about myself"

🌍 Real-world analogy
When someone asks "what's your name?", you say "My name is…" — my refers to yourself. In Apex, when a method or constructor refers to the current object's own variable, it uses this to be clear: "I mean this object's variable, not the parameter."
ThisKeyword.apex
public class Employee {

    public String  name;   // instance variable
    public Decimal salary;

    // ─────────────────────────────────────────────
    // Problem: constructor parameter has SAME name as the field
    // Without 'this', Apex doesn't know which 'name' you mean!
    // ─────────────────────────────────────────────
    public Employee(String name, Decimal salary) {
        // 'this.name' = the OBJECT's field
        // 'name'      = the CONSTRUCTOR's parameter
        this.name   = name;    // Assign parameter to the object's field
        this.salary = salary;
    }

    // ─────────────────────────────────────────────
    // 'this()' — call another constructor from within a constructor
    // Avoids duplicating setup code!
    // ─────────────────────────────────────────────
    public Employee(String name) {
        this(name, 30000);   // Calls the 2-param constructor above!
                              // Like saying: "do everything that one does,
                              // but use a default salary of 30,000"
    }

    public void applyRaise(Decimal percent) {
        // 'this.salary' is clear — the OBJECT's salary field
        this.salary = this.salary + (this.salary * percent / 100);
    }

    public void print() {
        System.debug('Employee: ' + this.name + ' | ₹' + this.salary);
    }
}

🔀 3. Method Overloading — "Same Name, Different Jobs"

🌍 Real-world analogy
A chef at a restaurant has one skill: cook(). But they cook differently depending on what you order — cook(pasta), cook(pizza), cook(salad). The name is the same, but the ingredients and result differ. That's overloading: same method name, different parameter signatures.

Overloading = same method name, different number or type of parameters. Apex figures out at compile time which version to call based on what you pass in. This is also called Compile-Time Polymorphism.

MethodOverloading.apex
public class MathHelper {

    // ─────────────────────────────────────────────
    // add() — Version 1: two integers
    // ─────────────────────────────────────────────
    public static Integer add(Integer a, Integer b) {
        System.debug('add(Integer, Integer) called');
        return a + b;
    }

    // ─────────────────────────────────────────────
    // add() — Version 2: three integers
    // Same name! Different number of params
    // ─────────────────────────────────────────────
    public static Integer add(Integer a, Integer b, Integer c) {
        System.debug('add(Integer, Integer, Integer) called');
        return a + b + c;
    }

    // ─────────────────────────────────────────────
    // add() — Version 3: two Decimals
    // Same name! Different TYPE of params
    // ─────────────────────────────────────────────
    public static Decimal add(Decimal a, Decimal b) {
        System.debug('add(Decimal, Decimal) called');
        return a + b;
    }

    // ─────────────────────────────────────────────
    // describe() — Version 1: no params
    // ─────────────────────────────────────────────
    public static String describe() {
        return 'I am a Math Helper!';
    }

    // describe() — Version 2: with a prefix String
    public static String describe(String prefix) {
        return prefix + ': I am a Math Helper!';
    }

    // describe() — Version 3: with prefix + version number
    public static String describe(String prefix, Integer version) {
        return prefix + ' v' + version + ': I am a Math Helper!';
    }
}

System.debug( MathHelper.add(2.5, 3.1) );      // calls version 3 → 5.6

System.debug( MathHelper.describe() );
System.debug( MathHelper.describe('INFO') );
System.debug( MathHelper.describe('INFO', 2) );

Overloading vs Overriding — the KEY difference:
Overloading = Same class, same method name, different parameters. Decided at compile time.
Overriding = Parent + child class, same method name, same parameters. Decided at runtime.

public class BankVault {

    // private — ONLY this class can see/change this
    // Nobody from outside can do: vault.balance = 999999;
    // They MUST go through the deposit/withdraw methods
    private   Decimal balance;
    private   String  ownerPin;
    protected String  ownerName;   // subclasses can see this

    // public constructor — anyone can create a vault
    public BankVault(String name, String pin, Decimal startAmount) {
        this.ownerName = name;
        this.ownerPin  = pin;       // stored privately, never exposed
        this.balance   = startAmount;
    }

    // public method — controlled way to deposit
    // Validates: amount must be > 0
    public void deposit(Decimal amount) {
        if (amount <= 0) {
            System.debug('ERROR: Deposit amount must be positive!');
            return;
        }
        balance += amount;
        System.debug(ownerName + ' deposited ₹' + amount
                   + '. New balance: ₹' + balance);
    }

    // public method — controlled withdrawal
    // Validates: sufficient funds AND correct PIN
    public Boolean withdraw(Decimal amount, String pin) {
        if (!isValidPin(pin)) {       // calls our private helper
            System.debug('ERROR: Wrong PIN!');
            return false;
        }
        if (amount > balance) {
            System.debug('ERROR: Insufficient funds!');
            return false;
        }
        balance -= amount;
        System.debug('Withdrew ₹' + amount + '. Remaining: ₹' + balance);
        return true;
    }

    // public read-only balance check (no PIN needed to see balance)
    public Decimal getBalance() {
        return balance;     // controlled read — no write allowed
    }

    // PRIVATE helper — only used internally, NOT accessible outside
    // No one outside can call vault.isValidPin() directly
    private Boolean isValidPin(String input) {
        return ownerPin == input;
    }
}

🎭 5. Polymorphism — "One Interface, Many Forms"

🌍 Real-world analogy
You press the Play button. On your music app it plays a song. On a video app it plays a movie. On a podcast app it plays an episode. Same button (same method call), completely different behaviour depending on what object it's called on. That's polymorphism.
PolymorphismDeepDive.apex
// ─────────────────────────────────────────────
// TYPE 1: RUNTIME POLYMORPHISM (via Override)
// The method to call is decided at RUNTIME, not compile time
// ─────────────────────────────────────────────

public virtual class Notification {
    public String  recipient;
    public String  message;

    public Notification(String to, String msg) {
        this.recipient = to;
        this.message   = msg;
    }

    // Virtual — each child will override with its own delivery method
    public virtual void send() {
        System.debug('Sending notification to: ' + recipient);
    }

    // NOT overridden — shared by all
    public void logAttempt() {
        System.debug('[LOG] Notification attempt for: ' + recipient);
    }
}

public class EmailNotification extends Notification {
    public String subject;

    public EmailNotification(String to, String sub, String msg) {
        super(to, msg);
        this.subject = sub;
    }

    public override void send() {
        System.debug('📧 EMAIL → To: ' + recipient
                   + ' | Subject: ' + subject
                   + ' | Body: '    + message);
    }
}

public class SMSNotification extends Notification {
    public String phoneNumber;

    public SMSNotification(String phone, String msg) {
        super(phone, msg);
        this.phoneNumber = phone;
    }

    public override void send() {
        System.debug('📱 SMS → To: +91-' + phoneNumber
                   + ' | Text: '  + message);
    }
}

public class PushNotification extends Notification {
    public PushNotification(String userId, String msg) {
        super(userId, msg);
    }

    public override void send() {
        System.debug('🔔 PUSH → UserID: ' + recipient
                   + ' | Alert: '  + message);
    }
}

// THE POWER OF POLYMORPHISM:
// Put all different notification types in ONE list of the PARENT type.
// Call send() on each — Apex automatically runs the RIGHT version!
// You don't need to know which type each one is.
// ─────────────────────────────────────────────

🔄 6. Type Casting — Changing the Variable's "Label"

🌍 Real-world analogy
You have a box labelled "Animal." Inside it is actually a Dog. You know it's a Dog, but the box label says Animal. Casting is relabelling the box to say "Dog" so you can use Dog-specific features like fetch().
TypeCasting.apex
// Using Animal/Dog/Cat from our inheritance section

// Widening cast (Upcasting) — always safe, done automatically
// Dog IS-A Animal, so storing a Dog in an Animal variable is fine
Animal a = new Dog('Bruno', 3, 'Labrador');
System.debug(a.speak());     // Woof! (Dog's version — polymorphism)
// a.fetch();  ← COMPILE ERROR! Animal type has no fetch()

// Narrowing cast (Downcasting) — need explicit cast to use child methods
// You're telling Apex: "trust me, this Animal is actually a Dog"
Dog d = (Dog) a;        // Cast Animal → Dog using (Dog) syntax
d.fetch();             // ✅ Now you can use Dog-specific method!

// SAFE casting with instanceof — check before casting!
// Prevents ClassCastException at runtime
List<Animal> animals = new List<Animal>{
    new Dog('Bruno', 3, 'Lab'),
    new Cat('Mimi', 2),
    new Dog('Max', 5, 'Pug')
};

for (Animal animal : animals) {
    // instanceof = "is this object actually a Dog?"
    if (animal instanceof Dog) {
        Dog dog = (Dog) animal;   // Safe to cast — we just checked!
        System.debug(dog.name + ' is a dog → ');
        dog.fetch();
    } else if (animal instanceof Cat) {
        System.debug(animal.name + ' is a cat → meow!');
    }
}

// Primitive type casting

🎯 7. Abstraction — "Show the What, Hide the How"

🌍 Real-world analogy
You use a washing machine: you press "Cotton Wash" and walk away. You don't care about water temperature algorithms or drum rotation speed. Abstraction = showing the user only what they need to interact with, hiding the complex internals.
AbstractionDemo.apex
// The abstract class defines WHAT needs to happen
// Each child decides HOW it happens
public abstract class PaymentGateway {

    public String  gatewayName;
    public Boolean sandboxMode;

    // ABSTRACT methods — WHAT to do (no body)
    // Every payment gateway MUST implement these differently
    public abstract Boolean processPayment(Decimal amount);
    public abstract String  getTransactionId();

    // CONCRETE method — shared HOW (already implemented)
    // Works the same for ALL gateways
    public void initiatePayment(Decimal amount) {
        System.debug('=== ' + gatewayName + ' Payment ===   ');
        System.debug('Mode: ' + (sandboxMode ? 'TEST' : 'LIVE'));
        Boolean success = processPayment(amount);  // calls child's version!
        if (success) {
            System.debug('✅ Payment success! TxnID: ' + getTransactionId());
        } else {
            System.debug('❌ Payment failed!');
        }
    }
}

// Razorpay implementation
public class RazorpayGateway extends PaymentGateway {
    public RazorpayGateway() {
        gatewayName = 'Razorpay';
        sandboxMode = true;
    }
    public override Boolean processPayment(Decimal amount) {
        System.debug('Razorpay: Charging ₹' + amount + ' via UPI/Card/Net Banking');
        return true;    // simulating success
    }
    public override String getTransactionId() {
        return 'RZP_' + Math.random().toString().left(8);
    }
}

// PayPal implementation — same interface, totally different internals
public class PayPalGateway extends PaymentGateway {
    public PayPalGateway() {
        gatewayName = 'PayPal';
        sandboxMode = false;
    }
    public override Boolean processPayment(Decimal amount) {
        System.debug('PayPal: Processing $' + (amount/83) + ' via PayPal account');
        return true;
    }
    public override String getTransactionId() {
        return 'PAYPAL_' + DateTime.now().getTime();
    }
}

// They do NOT count toward governor limits!
// They run in an isolated transaction with a fresh database
@isTest
public class HelloSalesforceTest {

    // @TestSetup runs once before ALL test methods — for shared setup data
    @TestSetup
    static void setupData() {
        // Create test accounts that ALL test methods can use
        List<Account> testAccs = new List<Account>{
            new Account(Name = 'Test Company A'),
            new Account(Name = 'Test Company B')
        };
        insert testAccs;

    // Name it descriptively: test_whatYoureTestingAnd_expectedResult
    @isTest
    static void test_greet_shouldPrintMessage() {
        // Test.startTest() / stopTest() marks the boundaries of your test
        // Any limits used BEFORE startTest don't count toward your class's limits
        Test.startTest();
            HelloSalesforce.greet('TestUser');   // Call your method
        Test.stopTest();

        // System.assert() — verify the expected result
        // If the assertion fails, the test FAILS and shows an error
        System.assert(true, 'greet() ran without error');

        Test.startTest();
            List<Account> result = HelloSalesforce.getTopAccounts(2);

        System.assertNotEquals(null, result, 'Result should not be null');

    }

📦 Inner Classes, Nested Classes & Enums

🌍 Real-world analogy
A car has an engine inside it. The engine isn't a separate entity on the street — it only makes sense inside a car. Inner classes are the same: a class defined inside another class because it only makes sense in that context. You use them to logically group helper structures together.

🔷 Inner Classes

InnerClassDemo.apex
// Outer class — the "wrapper"
public class ShoppingCart {

    public String customerName;
    private List<CartItem> items = new List<CartItem>();

    // ─────────────────────────────────────────────
    // INNER CLASS — only makes sense inside ShoppingCart
    // Represents a single line item in the cart
    // ─────────────────────────────────────────────
    public class CartItem {
        public String  productName;
        public Integer quantity;
        public Decimal unitPrice;

        public CartItem(String name, Integer qty, Decimal price) {
            this.productName = name;
            this.quantity    = qty;
            this.unitPrice   = price;
        }

        // Method on the inner class
        public Decimal lineTotal() {
            return quantity * unitPrice;
        }
    }

    // Outer class constructor
    public ShoppingCart(String name) {
        this.customerName = name;
    }

    // Add item to cart
    public void addItem(String product, Integer qty, Decimal price) {
        items.add(new CartItem(product, qty, price));   // Using inner class
    }

    // Calculate total
    public Decimal getTotal() {
        Decimal total = 0;
        for (CartItem item : items) {
            total += item.lineTotal();
        }
        return total;
    }

    // Print receipt
    public void printReceipt() {
        System.debug('=== Receipt for: ' + customerName + ' ===');
        for (CartItem item : items) {
            System.debug(item.productName + ' x' + item.quantity
                       + '  →  ₹' + item.lineTotal());
        }
        System.debug('TOTAL: ₹' + getTotal());
    }
}

🔶 Enums — A Fixed List of Named Constants

🌍 Real-world analogy
Traffic lights can only be RED, YELLOW, or GREEN — nothing else. If someone passes the value "PURPLE" to your traffic light code, that's a bug. An enum locks in the allowed values at compile time. No typos, no invalid values.
EnumDemo.apex
// ─────────────────────────────────────────────
// ENUM — define a fixed set of named values
// Think of it as a type-safe list of allowed options
// ─────────────────────────────────────────────
public enum OrderStatus {
    PENDING,
    PROCESSING,
    SHIPPED,
    DELIVERED,
    CANCELLED
}

public enum Priority { LOW, MEDIUM, HIGH, CRITICAL }

public class Order {
    public String      orderId;
    public OrderStatus status;
    public Priority    priority;

    public Order(String id) {
        this.orderId  = id;
        this.status   = OrderStatus.PENDING;    // default
        this.priority = Priority.MEDIUM;
    }

    public void ship() {
        // Switch on enum values — clean and readable!
        if (status == OrderStatus.PROCESSING) {
            status = OrderStatus.SHIPPED;
            System.debug(orderId + ' has been shipped! 📦');
        } else {
            System.debug('Cannot ship — current status: ' + status);
        }
    }

    public void printStatus() {
        System.debug('Order: ' + orderId
                   + ' | Status: '   + status
                   + ' | Priority: ' + priority);
    }
}

💥 Exception Handling — When Things Go Wrong

🌍 Real-world analogy
You're driving (your code is running). Suddenly a tyre punctures (an error occurs). You have two choices: ignore it and crash the car (unhandled exception — your whole transaction fails), or pull over safely and call roadside assistance (try-catch — handle the error gracefully and continue).

Exception Hierarchy — Know Your Enemy

Exception TypeWhen It Happens
DmlExceptionDML fails — required field missing, duplicate rule, etc.
QueryExceptionSOQL returns 0 rows but you assigned to single object; or returns multiple rows
NullPointerExceptionYou call a method on a null variable
LimitExceptionGovernor limit exceeded (cannot be caught!)
MathExceptionDivision by zero
StringExceptionInvalid string operation
TypeExceptionInvalid type cast
ExceptionThe parent of all — catches anything
ExceptionHandling.apex
// ─────────────────────────────────────────────
// BASIC TRY-CATCH-FINALLY
// ─────────────────────────────────────────────
try {
    // Code that might fail goes here
    Account acc = [SELECT Id FROM Account WHERE Name = 'DOES_NOT_EXIST'];
    // ↑ This throws QueryException — no rows found!

} catch (QueryException qe) {
    // Specific catch — only for query errors
    System.debug('Query failed: ' + qe.getMessage());

} catch (DmlException de) {
    // Specific catch — only for DML errors
    System.debug('DML failed: ' + de.getMessage());

} catch (Exception e) {
    // Generic catch — catches ANYTHING not caught above
    // Always put this LAST — it's the widest net
    System.debug('Unexpected error: ' + e.getMessage());
    System.debug('Error type: '       + e.getTypeName());
    System.debug('Stack trace: '      + e.getStackTraceString());

} finally {
    // ALWAYS runs — whether exception happened or not
    // Good for cleanup: close connections, log to a custom object, etc.
    System.debug('=== Execution completed ===');
}

// ─────────────────────────────────────────────
// NULL POINTER — most common beginner mistake!
// ALWAYS check for null before calling methods on an object
// ─────────────────────────────────────────────
Account maybeNull = null;

try {
    System.debug(maybeNull.Name);  // BOOM — NullPointerException!

// ─────────────────────────────────────────────
// Create your OWN exception class — must extend Exception
// Name convention: always end with "Exception"
// ─────────────────────────────────────────────
public class InsufficientStockException extends Exception {}
public class InvalidAgeException       extends Exception {}
public class PaymentDeclinedException   extends Exception {}

public class Inventory {
    private Map<String, Integer> stock = new Map<String, Integer>{
        'Laptop'   => 5,
        'Phone'    => 12,
        'Headphone'=> 3
    };

    public void sell(String item, Integer qty) {
        if (!stock.containsKey(item)) {
            // Throw custom exception with a clear message
            throw new InsufficientStockException(
                'Product not found: ' + item);
        }
        if (stock.get(item) < qty) {
            throw new InsufficientStockException(
                'Only ' + stock.get(item)
                + ' of "' + item + '" in stock. Requested: ' + qty);
        }
        stock.put(item, stock.get(item) - qty);
        System.debug('Sold ' + qty + 'x ' + item
                   + '  →  Remaining: ' + stock.get(item));
    }
}

Inventory inv = new Inventory();

try {
    inv.sell('Laptop', 3);          // ✅ OK — 5 in stock
    inv.sell('Headphone', 5);       // ❌ Only 3 in stock!
} catch (InsufficientStockException ise) {
    System.debug('Stock error: ' + ise.getMessage());
}

try {
    inv.sell('Tablet', 1);          // ❌ Product doesn't exist!
} catch (InsufficientStockException ise) {
    System.debug('Stock error: ' + ise.getMessage());

⚡ Apex Triggers — Code That Runs on Record Changes

🌍 Real-world analogy
A trigger is like a motion-sensor light in a room: when something happens (someone enters = record is inserted/updated/deleted), the light automatically turns on (your code runs). You don't press any button — it fires on its own in response to a database event.
EventWhen it fires
before insertBEFORE a new record is saved — use to validate or set default field values
after insertAFTER a new record is saved — use to create related records (Id is available)
before updateBEFORE an existing record is changed — use to validate changes
after updateAFTER an existing record is changed — use to update related records
before deleteBEFORE a record is deleted — use to prevent deletion or archive
after deleteAFTER a record is deleted
after undeleteAFTER a record is restored from Recycle Bin
AccountTrigger.trigger
// ─────────────────────────────────────────────
// A trigger lives in /force-app/main/default/triggers/
// File is .trigger, not .cls
// Syntax: trigger TriggerName on ObjectName (events)
// ─────────────────────────────────────────────
trigger AccountTrigger on Account (before insert, before update,
                                     after insert,  after update) {
    // Best practice: Triggers should be THIN — just call a handler class.
    // Never put business logic directly inside the trigger itself!
    AccountTriggerHandler.handle(Trigger.new, Trigger.old,
                                      Trigger.newMap, Trigger.oldMap,
                                      Trigger.isInsert, Trigger.isUpdate,
                                      Trigger.isBefore, Trigger.isAfter);
}
AccountTriggerHandler.cls — The Handler Class
// ─────────────────────────────────────────────
// TRIGGER CONTEXT VARIABLES — automatically available in every trigger
//
// Trigger.new     → List of NEW records (insert/update)
// Trigger.old     → List of OLD records before change (update/delete)
// Trigger.newMap  → Map of new records
// Trigger.oldMap  → Map of old records
// Trigger.isInsert / isUpdate / isDelete / isUndelete
// Trigger.isBefore / isAfter
// Trigger.size    → how many records are being processed
// ─────────────────────────────────────────────

public with sharing class AccountTriggerHandler {

    public static void handle(
        List<Account> newList,
        List<Account> oldList,
        Map<Id, Account> newMap,
        Map<Id, Account> oldMap,
        Boolean isInsert,
        Boolean isUpdate,
        Boolean isBefore,
        Boolean isAfter
    ) {
        if (isBefore && isInsert)  beforeInsert(newList);
        if (isBefore && isUpdate)  beforeUpdate(newList, oldMap);
        if (isAfter  && isInsert)  afterInsert(newList);
        if (isAfter  && isUpdate)  afterUpdate(newList, oldMap);
    }

    // ── BEFORE INSERT: set default values before saving ──
    private static void beforeInsert(List<Account> newAccounts) {
        for (Account acc : newAccounts) {
            // Auto-set Rating to 'Warm' if no Rating provided
            if (acc.Rating == null) {
                acc.Rating = 'Warm';
            }
            // Standardise phone format
            if (acc.Phone != null) {
                acc.Phone = acc.Phone.replaceAll([^0-9], '');
            }
        }
    }

    // ── BEFORE UPDATE: compare old vs new, reject invalid changes ──
    private static void beforeUpdate(List<Account> newList,
                                        Map<Id, Account> oldMap) {
        for (Account newAcc : newList) {
            Account oldAcc = oldMap.get(newAcc.Id);   // get old version

            // Business rule: don't allow removing a phone number once set
            if (oldAcc.Phone != null && newAcc.Phone == null) {
                // addError() blocks the save and shows a user-friendly message
                newAcc.addError('Cannot remove a phone number once it is set.');
            }

            // Detect if Rating actually changed (avoid processing unchanged records)
            if (newAcc.Rating != oldAcc.Rating) {
                System.debug('Rating changed from ' + oldAcc.Rating
                           + ' to ' + newAcc.Rating);
            }
        }
    }

    // ── AFTER INSERT: create related records (Id is now available) ──
    private static void afterInsert(List<Account> newAccounts) {
        List<Contact> defaultContacts = new List<Contact>();
        for (Account acc : newAccounts) {
            // Auto-create a default contact for every new account
            defaultContacts.add(new Contact(
                LastName  = 'Primary Contact',
                AccountId = acc.Id    // acc.Id is available because we're after insert
            ));
        }
        if (!defaultContacts.isEmpty()) {
            insert defaultContacts;
        }
    }

    // ── AFTER UPDATE: sync changes to related records ──
    private static void afterUpdate(List<Account> newList,
                                       Map<Id, Account> oldMap) {
        Set<Id> changedIds = new Set<Id>();
        for (Account acc : newList) {
            // Only process accounts where Name actually changed
            if (acc.Name != oldMap.get(acc.Id).Name) {
                changedIds.add(acc.Id);
            }
        }
        System.debug('Accounts with changed names: ' + changedIds.size());
    }
}
